# PITFALLS.md — qmd-engine 避坑录

这份文档是这个包的灵魂：不是 API 文档，是踩过的坑。工程经验，不讲私事。

---

## 1. chunk 尺寸必须小于嵌入模型有效 token 窗口

**踩过的坑**：chunk 切到几千字符量级，远超嵌入模型实际有效的 token 窗口
（哪怕模型标称支持更大的 `n_ctx`）。超窗口的文本被模型截断或以退化方式
处理，产出的向量质量大幅劣化——检索结果和随机排序差别不大，但**不报
任何错误**，只是"搜出来的东西对不上"。

**根因**：chunk 长度的设计目标是"塞进模型上下文窗口后还能被有效编码"，
不是"塞得下就行"。一个 4500 字符的中文 chunk，按中文平均 1.5-2 字符/token
估算，可能到 2000+ token；如果模型有效编码窗口只有几百 token，后面大半段
文本对最终向量几乎没有贡献。

**修法**：
- chunk 目标长度留在模型 `n_ctx` 以内，并留出安全余量（本包 `search_embed.py`
  用 `n_ctx=8192`，`full_scan.py` 的 `_chunk_text_generic` 按约 85 "token"
  ×3 字符估算切块，代码 chunk 按 50 行切）
- 换模型/换 chunk 策略后，**必须重新验证**：拿几条已知答案的 query 跑一遍
  `/search`，看 top1 是不是预期结果，而不是只看"跑通了不报错"

## 2. 换 embedding 模型必须全量重算索引

**踩过的坑**：索引里混了两种不同维度/不同模型产出的向量（比如旧模型
512 维、新模型 2560 维），`embeddings @ query_vec` 这类矩阵运算要么直接
维度不匹配报错，要么在某些实现里静默产出错误分数——取决于代码有没有
提前校验。这类污染索引一旦写盘，之后每次查询都在算无意义的结果。

**根因**：向量空间是模型专属的。不同模型（哪怕架构相似）产出的向量
不在同一个空间里，不能混合检索、不能直接比较余弦相似度。

**修法**：
- 用一个唯一配置出处记录当前模型名 + 期望维度（本包 `qmd_config.py` 的
  `EXPECTED_EMBED_DIM` / `MODEL_NAME`），写盘前后都校验
- 写盘前的最后一道闸：`full_scan.py` 的 `validate_embedding_dim()`，
  维度不对直接 `sys.exit(3)` 拒绝写盘，不静默降级
- daemon 加载索引时也校验一遍（双保险，因为索引可能是被别的路径写的）
- 换模型 = 必须全量 `rebuild`，没有"部分复用旧向量"这种选项

## 3. 索引重建必须原子化

**踩过的坑**：重建进程被杀（OOM/手动中断/机器重启）在写盘中途，留下
半成品索引文件。轻则索引损毁需要从备份恢复，重则**损毁两天没人发现**——
因为程序读到了半成品文件也没报错，只是返回的结果范围变小了。

**根因**：直接对着正式索引目录原地写、原地覆盖，中途失败就是不可逆的
"写了一半"状态；没有任何机制能区分"这是完整索引"还是"这是被打断的半成品"。

**修法**（`full_scan.py` 的 `rebuild_index()`）：
1. 新索引写到一个空的 staging 目录（不碰旧索引一根手指）
2. staging 里的产物做自洽校验（chunks 和 embeddings 都必须存在且行数对齐）
3. 校验通过后，才用 `os.replace()`（同文件系统内的原子操作）把 staging
   里的文件逐个顶替进正式索引目录，旧版本转存为 `.bak-pre-rebuild-<时间戳>`
4. 任何一步失败：staging 半成品整个作废（`shutil.rmtree`），旧索引全程
   原封不动、可用

`os.replace()` 是这里的关键——它在同一文件系统内是原子操作，不会有
"文件被删了但新文件还没完全写入"的中间状态。

## 4. 增量写盘要事务化，禁止"截断和稀泥"式容错

**踩过的坑**：早期实现里，发现 chunks.jsonl 和 embeddings.npy 行数对不上
时，选择"截断到较小的那个数，凑合继续跑"。结果一次写盘异常 + 这种截断
逻辑组合，把十几万条的索引静默砍到几千条，而且因为 file_index.json 仍然
记着"这些文件已经索引过"，后续的增量扫描也不会补回来——**索引损毁进
了一种自我巩固的稳定状态**，直到人工发现搜索结果少得离谱才追查出来。

**根因**：截断是一种隐式的"数据丢了就丢了，继续跑"决策，把一次异常
悄悄转换成了永久性的数据丢失，而且不留任何显式痕迹。

**修法**：
- 三件套（chunks / embeddings / file_index）行数不一致时，**拒绝继续**，
  写一个 `NEEDS-REBUILD` 标记文件说明原因，`sys.exit(3)`，等人工介入
- 加一道"缩水保险"：增量流程如果算出来的最终 chunk 数比运行前少了超过
  50%，大概率是加载了残本/挂载盘没接上之类的事故，同样拒绝写盘
- 真正落盘前，把最耗内存/最容易被 OOM 打断的序列化步骤（`np.save` 大
  数组）提前到内存里做完（`BytesIO`），再连续走几个 `os.replace()`——
  把"数据不一致的时间窗口"从"整个嵌入耗时"压缩到几个原子操作的毫秒级

## 5. 检索失败必须显式区分"无结果"和"查询失败"

**踩过的坑**：daemon 返回 HTTP 200 加一个空结果列表，调用方分不清是
"这个 query 真的没有相关内容"还是"后端哪里坏了返回了空的"。健康自检
如果不特意区分这两种情况，一个真实的静默故障会被误读成"正常，只是
没搜到"。

**修法**：
- `health_check.py` 的探针词故意选**已知命中**的词（比如索引库里
  一定存在的关键词），如果这种"必中"探针返回 `count=0`，直接判定为
  故障（`"count=0（daemon 200 但空 · 静默失败模式）"`），而不是当作
  正常的零结果
- 三层（grep / `/search_fast` / `/search`）分别探测，任一层失败都单独
  报出来，不要合并成一个笼统的"健康/不健康"

## 6. 三层路由哲学：按能力边界选层，不是按顺序全跑

不要每次查询都把 grep + embedding + reranker 全跑一遍，也不要盲目
只依赖最贵的那一层。三层的定位不同：

| 层 | 成本 | 适用场景 |
|---|---|---|
| L1 grep 精确匹配 | 免费、零延迟 | 知道确切关键词/文件名时优先用 |
| L2 embedding 召回（`/search_fast`） | 本地算力，快（~0.4s 量级） | 关键词说不准、模糊语义查询的默认通道 |
| L3 reranker 精排（`/search`） | 慢（秒级，跑一次 cross-encoder） | L2 结果不够好时的最后手段，不作为默认路径 |

按查询的性质选层，而不是无脑走全套三层——多数查询 L1 或 L2 就能满足，
把 L3 留给真正需要精排的场景，能显著降低平均延迟。

## 7. 查询侧 instruct 前缀对召回质量的影响

Qwen3-Embedding 系列模型区分"非对称检索"场景：查询（query）和文档
（document/passage）用不同的编码方式更合适。本包在 `embedding_search()`
里给查询文本加了官方推荐的 instruct 前缀（"Instruct: Given a web search
query, retrieve relevant passages that answer the query\nQuery: "），
但**文档侧 `embed_texts()`（索引构建 / daemon `/embed` 用的那个函数）
绝不能加这个前缀**——一旦两边都加，或者两边都不加，召回质量都会明显
下降，因为破坏了模型训练时假设的"不对称"结构。

这是个容易踩的坑，因为代码层面两个函数长得很像，复制粘贴时很容易
把前缀带错地方。改动这块代码前，先跑几条已知答案的 query 做回归验证。
