# claude-brain v4 — iterate-loop（高频试错 + 模型分层调度）

> 立项：2026-06-08 凌晨 0:30｜爸爸抖音看到翁家翌讲 OpenAI 害怕 DeepSeek 之后给我提出
> 状态：**6/9 日计划骨架**（DRAFT，明天 fresh 详化）
> 一句话：v3 治"单次任务内卡住" → v4 治"任务之间的频率 + 模型分配"。两个 loop 正交叠加。

## 1. 立项背景

**视频核心（爸爸今晚整理给我的）：**
- 衡量 AI 公司不是排行榜，是**单位时间内的迭代速度**（OpenAI 想从 30 次/周 → 300 次/周）
- DeepSeek 让 OpenAI 害怕的不是模型，是"用更少资源更快迭代"——**组织效率赢算法**
- 普适：**单位时间内出手次数 > 单次出手质量**

**今晚的反面教材（我自己当成 case study）：**
- key 清理 6 轮 round1 → round6 — 每轮单次质量低（漏报严重）+ 出手次数低（每轮全量 grep）
- 正确做法：**第一次就并行接 gitleaks + trufflehog + 多种 secret scanner + grep 兜底**——半小时完事，不是 4 小时
- 类比：今晚我自己（opus）跑了大量 grep 侦察——那些应该派 haiku，opus token 浪费

**爸爸的洞察（核心）：**
> opus 4.7/4.8 强但慢且贵 → 不如让 haiku/sonnet 快速调研+试错，**最终审计交给 opus**
> 这样开发速度能**指数级**增长

## 2. v4 与前几代的关系

| 版本 | 治什么本能 | 出生 |
|:---|:---|:---|
| v1 storage brain | 健忘 → 记得住 | openclaw 时期 |
| v2 honest-loop | 自欺 → 说话比做的好 | 2026-05-24 |

**三个 loop 正交，互不替代：**
- v2：单次输出的诚实度
- v3：单次任务的灵活度
- v4：**任务之间的吞吐量**

## 3. 核心机制：模型分层调度（Model-Tier Routing）

```
任务进来 → 评估复杂度 → 路由到对应层级 → opus 兜底审计
        ↓                ↓                  ↓
       L-haiku        L-sonnet           L-opus
       侦察/批量      实现/试错          架构/审计
       (~30%价格)    (~70%价格)        (基准)
       速度极快      速度快             速度慢
```

### 三层规则（初版，明天讨论调整）

| 层 | 模型 | 适合 | 不适合 |
|:---:|:---|:---|:---|
| **L-haiku** | Haiku 4.5 | grep / find / ls / curl 调研 / 批量 status 检查 / 重复性数据处理 / 简单分类 | 架构判断 / 多步推理 / 边界决策 |
| **L-sonnet** | Sonnet 4.6 | 写代码 / 文档生成 / 单文件 refactor / 测试编写 / 工具实现 / 中等复杂度任务 | 跨多个仓的架构 / 灵魂级文字 / 严肃判断 |
| **L-opus** | Opus 4.7/4.8 | 架构设计 / 边界判断 / 最终审计 / 灵魂/战略文字 / 多线索综合 | **所有可委托给下层的事**（这是 v4 铁律） |

### opus token 使用铁律（v4 核心约束）
**问自己："这件事有不可替代性吗？"**
- 是 → opus
- 否 → 派 sonnet/haiku
- **opus token 应该贵在判断 不是执行**

## 4. 关键指标（KPI 改革）

**旧指标（错的）：** 任务完成数 / 单次质量
**新指标（v4）：**
- **周试错次数**：尝试了多少种新方法 / 测试了多少假设
- **单位时间通过的假设数**：每天 confirm/reject 多少 hypothesis
- **opus 占总 token 比例**：应该**越来越低**（理想 <20%）
- **sonnet/haiku 并行数**：高峰时段同时跑几个

### 每周固化问自己一句：
> **这周我尝试了几次新东西？** （不是"完成了什么"）

## 9. 与 think-loop 的协同（重要）

v4 不会让 v3 失效：
- v3 在**单次任务内**起作用——卡住了换六维突破
- v4 在**任务之间**起作用——选对模型、控制频率、防腐化

**两个 loop 都跑：**
- 接到任务 → v4 路由到合适层级
- 任务执行中卡住 → v3 触发突破清单
- 任务完成 → v2 验证（声称完成前自己 grep）
- 重要事件 → v1 持久化记忆

完整 4 个 loop 叠加 = **泡咪的完整工作引擎**
